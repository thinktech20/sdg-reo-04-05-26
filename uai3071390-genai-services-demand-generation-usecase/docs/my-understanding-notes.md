## Q: What does "VS at sync time" mean? What is "VS auto-embed"?

### The short answer
"VS at sync time" means: Databricks Vector Search reads your text, converts it to a vector (embedding) by itself, and stores that vector in the index — you never touch the embedding. You just write plain text to a Delta table, and Databricks does the rest automatically when you trigger a sync.

### Longer explanation

**What is a Vector Search index?**
A VS index is like a search engine index but for meaning (semantics), not keywords. To search by meaning, every piece of text must be converted into a list of numbers called an **embedding vector**. When you search, your query is also converted to a vector, and VS finds the stored vectors closest to it.

**Two ways to get embeddings into the index:**

| Mode | Who generates the embedding? | When? | What you store in Delta? |
|---|---|---|---|
| **Delta Sync (auto-embed)** — what Alex uses | Databricks VS itself, using a model configured on the index | At sync time — when you call `triggerRefresh()` | Just the raw text (`chunk_text`) |
| **Self-managed index** — what the spec requires | Your pipeline code, using LiteLLM/Azure OpenAI | At ingest time — before you write to Delta | The text AND the pre-computed embedding array |

**"Sync time"** = the moment you call VS's `triggerRefresh()` / delta-sync API. VS wakes up, reads the new rows from your Delta table, calls its own internal embedding model on the `chunk_text` column, and stores the resulting vectors in the index. You're not in the loop.

**The model Databricks uses for auto-embed in Alex's POC:** `databricks-gte-large-en`
This is a GE-Vernova/Databricks-hosted model. It produces ~768-dimensional vectors.

**The model the spec wants:** `azure-text-embedding-3-large-1` via LiteLLM
This is GE's Azure OpenAI deployment. It produces 3072-dimensional vectors.

---

### Why is the mismatch a problem?

When you search:
1. Your query text gets converted to a vector using `azure-text-embedding-3-large-1` (3072 dims)
2. VS looks for the nearest stored vectors — but those were generated by `databricks-gte-large-en` (768 dims)

These are completely different number spaces. It's like asking "find me the closest point" but one set of coordinates is in miles and the other is in kilometers — the math runs but the answer is meaningless.

**Why does HYBRID search make it worse?**
HYBRID = keyword search (BM25) + dense vector search (embedding similarity) combined.
The keyword leg works fine. The vector leg is broken due to the model mismatch. So HYBRID in the current `fsr_pipeline` is effectively doing keyword search with a corrupted bonus signal.

**Why do the eval results (R@1=0.432) still look reasonable?**
Because those numbers come from `fsr_pipeline_gt_direct`, which uses a *self-managed* VS index (`vs_field_service_report_gt_litellm`) built with LiteLLM embeddings — both ingest and query use the same model. That pipeline is consistent. The main `fsr_pipeline` with its mismatched models has never been properly evaluated.

---

### What would "self-managed" look like in code?

In the current POC (Delta Sync):
```
1. chunk PDF → write chunk_text to Delta (no embedding)
2. call VS triggerRefresh()
3. VS reads chunk_text → calls databricks-gte-large-en internally → stores vectors
```

In the production spec (self-managed):
```
1. chunk PDF
2. call LiteLLM → get embedding vector for chunk_text
3. write chunk_text + embedding_vector to Delta
4. VS reads the pre-computed embedding_vector → stores it as-is (does NOT re-embed)
```

The production spec requires step 2 and the embedding column in Delta because it wants YOUR model (azure-text-embedding-3-large-1) to control embeddings, not Databricks's model.

## Q: What is Recall here? What is Recall@1, Recall@5?

### The short answer
Recall@k = "Did the right answer appear anywhere in the top k results?"
It is binary — either yes (1.0) or no (0.0). There is no partial credit.

### What is being tested?
For each (ESN, issue) pair in the ground truth, the evaluator asks:
> "If I search for this issue using this ESN, does the correct FSR chunk appear somewhere in the top k results?"

A "correct chunk" = a chunk from the right PDF, on roughly the right page (within ±10 pages), OR containing the right text snippet.

### How the code computes it (`evaluate_retrieval.py:L816-L821`)

```python
cumulative_matches = 0
for k in range(1, max_k + 1):
    if k <= len(match_flags) and match_flags[k - 1]:
        cumulative_matches += 1

    recall = 1.0 if cumulative_matches > 0 else 0.0
```

Walk through this step by step:

- VS returns up to 20 results (ranked #1 to #20)
- For each result, the code checks: is this the right chunk? → True or False (stored in `match_flags`)
- It then walks through k=1, 2, 3... up to 20
- At each k: "Have I seen at least one correct result in positions 1 through k?"
  - Yes → recall = 1.0
  - No → recall = 0.0

### Concrete example

Say VS returns these results for one query (✓ = correct chunk, ✗ = wrong):

```
Position 1: ✗
Position 2: ✗
Position 3: ✓  ← first correct result here
Position 4: ✗
Position 5: ✗
```

Then:
- Recall@1 = 0.0 (no correct result in top 1)
- Recall@2 = 0.0 (no correct result in top 2)
- Recall@3 = 1.0 (correct result found by position 3)
- Recall@4 = 1.0
- Recall@5 = 1.0

### What does the average mean?

The eval runs this for all 132 (ESN, issue) pairs and averages the recall scores.

So **Recall@1 = 0.432** means:
> For 43.2% of the 132 test queries, the correct chunk was the very first result returned.

**Recall@5 = 0.667** means:
> For 66.7% of queries, the correct chunk appeared somewhere in the top 5 results.

### Why does reranking hurt Recall@1?

Reranking reshuffles the top results using a different model (DatabricksReranker).
It is designed to improve overall quality, but it sometimes moves the correct chunk
*down* from position 1 to position 2 or 3.

So after reranking:
- Recall@1 drops (correct chunk got pushed down)
- Recall@5 stays the same (correct chunk is still in the top 5, just in a different position)

This is exactly what the eval shows:
```
hybrid_retrieval:  R@1=0.432, R@5=0.667  ← correct chunk often at #1
hybrid_reranking:  R@1=0.303, R@5=0.667  ← reranker moved it down, but still in top 5
```

The reranker is not "wrong" — it just optimises for something slightly different
than "put the single best result at position 1".

## Q: What is ANN in the eval results table?

ANN = **Approximate Nearest Neighbor**.

It is one of two ways to search a vector index:

| Mode | How it works | Trade-off |
|---|---|---|
| **ANN** | Finds vectors that are *approximately* closest to your query vector — uses shortcuts to skip most of the index | Faster, but might miss the single best match |
| **HYBRID** | Combines ANN (meaning-based) + BM25 keyword search, then merges the two ranked lists | Slower, but more robust — catches both semantic and keyword matches |

### Why does HYBRID beat ANN in the eval?

From the results:
```
hybrid_retrieval  R@1 = 0.432
ann_retrieval     R@1 = 0.379
```

FSR documents have specific terminology (ESN codes, event types, part numbers). Pure ANN can miss a chunk because the embedding isn't similar enough, but BM25 would catch it because the exact keyword is there. HYBRID gets the benefit of both.

### The four eval modes explained

| Mode | Search type | Reranking applied? |
|---|---|---|
| `ann_retrieval` | ANN only | No |
| `ann_reranking` | ANN only | Yes |
| `hybrid_retrieval` | ANN + BM25 | No |
| `hybrid_reranking` | ANN + BM25 | Yes |

The winner is `hybrid_retrieval` — no reranking, both search types combined.
