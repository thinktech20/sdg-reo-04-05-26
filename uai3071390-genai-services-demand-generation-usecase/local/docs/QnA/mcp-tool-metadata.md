# MCP Tool Naming ŌĆö Server-Side Canonical Names via `mcp_names`

## Problem Statement

The QnA agent receives MCP tool names auto-generated by FastMCP from OpenAPI operationIds.
These generated names are unstable for three compounding reasons:

1. **Double prefix** ŌĆö `assessments.router` declares its own prefix `/dataservices/api/v1/assessments`,
   and `temp_app` re-mounts it with `prefix="/dataservices/api/v1"`, producing operationIds like
   `get_findings_dataservices_api_v1_dataservices_api_v1_assessments_{assessment_id}_findings_get`.
2. **FastMCP 56-char truncation** ŌĆö `FastMCPOpenAPI._generate_default_name()` truncates tool names
   to 56 characters. Any operationId over 56 chars gets silently chopped, producing unstable suffixes.
3. **Brittle static alias table** ŌĆö The agent's `_TOOL_NAME_ALIASES` in `registry.py` must exactly
   match whatever truncated string comes out. Any path, prefix, or FastMCP version change breaks it.

This has caused repeated production issues where persona-facing tools (`read_re_table`, `read_re_report`)
silently disappear from the agent's tool set.

## FAQ: Why not rely on default MCP tool names?

> "The agent should understand MCP out of the box ŌĆö why do we need `mcp_names`?"

The agent **does** understand MCP out of the box. The problem isn't MCP ŌĆö it's the tool
**names** FastMCP auto-generates from our OpenAPI spec. Three things compound:

1. **Double prefix** ŌĆö Our FastAPI router mounting duplicates the path prefix in the
   operationId, producing names like
   `get_findings_dataservices_api_v1_dataservices_api_v1_assessments_{assessment_id}_findings_get`.
2. **56-char truncation** ŌĆö FastMCP silently chops anything over 56 characters, producing
   unstable suffixes that change if the path changes even slightly.
3. **Prompt mismatch** ŌĆö The system prompt tells the LLM to use `read_risk_matrix`, but the
   MCP server exposes something like `load_heatmap_dataservices_api_v1_da` (truncated).
   The LLM has no way to connect these.

Without `mcp_names`, the LLM sees garbled auto-generated names that don't match what's in
the system prompt. `mcp_names` is a **built-in FastMCP feature** (not a custom hack) that
maps operationIds to clean names *before* truncation happens. The old workaround ŌĆö a static
alias table on the agent side ŌĆö was strictly worse and caused repeated production outages.

## Chosen Solution: `mcp_names` parameter (FastMCP built-in)

FastMCP 2.14.4 `FastMCP.from_openapi()` accepts an `mcp_names: dict[str, str]` parameter that maps
operationId to a desired tool name. This lookup happens **before** slugification and truncation,
making it immune to all three problems above.

No vendor extensions (`x-displayName`) or agent-side metadata parsing needed.

### How it works

```
OpenAPI operationId ŌåÆ mcp_names lookup ŌåÆ canonical name ŌåÆ MCP tool name
                          Ōåæ                     Ōåæ
                    (our mapping)        (what LLM sees)
```

If an operationId is NOT in `mcp_names`, FastMCP falls back to its default behavior
(slugify + truncate to 56 chars). Non-persona-facing tools (health checks, internal routes)
don't need entries ŌĆö they are excluded by `route_maps` or simply not in the persona allowlist.

### How the LLM learns tool descriptions

`mcp_names` only controls the tool **name**. The tool **description** and **input schema**
come from the OpenAPI spec, which FastMCP passes through to the agent unchanged:

```
FastAPI endpoint docstring / description / Query(..., description=...)
  ŌåÆ OpenAPI spec (summary + description + parameters)
    ŌåÆ FastMCP.from_openapi() builds MCP tool schema
      ŌåÆ tools/list returns { name, description, inputSchema } to the agent
        ŌåÆ LLM sees all of this as the tool definition
```

The LLM gets tool descriptions from two complementary sources:

1. **OpenAPI spec ŌåÆ MCP tool schema** ŌĆö FastMCP packages each endpoint's docstring and
   parameter descriptions into the tool `description` and `inputSchema`. For example,
   `load_heatmap` has docstring `"Return risk-matrix rows for the given equipment type
   and persona."` and its Query params have descriptions like `"GEN or GT"`.

2. **System prompt** ŌĆö The RE/OE system prompts describe each tool in natural language
   (e.g., `read_risk_matrix: Query the Unit Risk Matrix to retrieve issue definitions...`).
   This gives the LLM higher-level guidance on **when** to use each tool.

Endpoints without docstrings (e.g., `get_ibat_equipment_endpoint`) still work ŌĆö
FastMCP generates a basic description from the endpoint path and parameters.

### Why this is robust

| Failure mode | Old approach (static aliases) | New approach (mcp_names) |
| --- | --- | --- |
| Route path changes | Breaks alias table | No impact ŌĆö operationId = Python function name |
| FastMCP truncation | Breaks alias table | No impact ŌĆö mcp_names lookup happens before truncation |
| Double prefix | Produces unrecognizable names | No impact ŌĆö prefix-based matching uses function name |
| New tools added | Must add exact truncated alias | Add one entry to `_CANONICAL_TOOL_NAMES` |
| FastMCP version change | Truncation length may change | No impact ŌĆö mcp_names bypasses auto-naming |

## Canonical Tool Name Map

Each entry maps a Python function name prefix (stable, under our control) to the prompt-facing
canonical name used in system prompts and persona allowlists.

### Server-side: `_CANONICAL_TOOL_NAMES` in `data_service/mcp/server.py`

```python
# Canonical prompt-facing names for persona-relevant MCP tools.
# Keys are Python function name prefixes (stable identifiers we control).
# Values are the names the LLM sees in system prompts.
#
# Confirm with: RE prompt (re_system_prompt.txt), OE prompt (oe_system_prompt.txt),
#               and persona allowlists in registry.py _PERSONA_TOOLS.
_CANONICAL_TOOL_NAMES: dict[str, str] = {
    "get_ibat_equipment_endpoint": "read_ibat",          # GET /ibat/equipment
    "read_prism":                  "read_prism",          # POST /prism/read
    "load_heatmap":                "read_risk_matrix",    # GET /heatmap/load
    "get_er_cases_endpoint":       "query_er",            # GET /er/cases
    "get_risk_er_cases_endpoint":  "query_risk_er",       # POST /er/risk-er-cases
    "retrieve_endpoint":           "query_fsr",           # POST /retriever/retrieve
    "get_findings":                "read_re_table",       # GET /assessments/{id}/findings
    "get_assessment_by_id":        "read_re_report",      # GET /assessments/{id}
    "get_risk_analysis":           "read_risk_analysis",  # GET /assessments/{id}/risk-analysis
}
```

**Note on `read_re_report` vs `read_risk_analysis`:**
Previously both `get_assessment_by_id` and `get_risk_analysis` were aliased to `read_re_report`.
FastMCP's `_get_unique_name` would silently rename the second to `read_re_report_2`.
To avoid this, `get_risk_analysis` now has its own canonical name `read_risk_analysis`.
The persona allowlists and system prompts should be updated to include `read_risk_analysis`
as a separate tool.

### Server-side: `_build_mcp_names()` function

Scans the OpenAPI spec for operationIds and matches them by function-name prefix:

```python
def _build_mcp_names(openapi_spec: dict) -> dict[str, str]:
    """Map auto-generated operationIds to canonical prompt-facing names.

    Matches by function-name prefix so that path changes, double prefixes,
    and slug suffixes don't affect the mapping.
    """
    mcp_names = {}
    for path_methods in openapi_spec.get("paths", {}).values():
        for method_detail in path_methods.values():
            if not isinstance(method_detail, dict):
                continue
            op_id = method_detail.get("operationId")
            if not op_id:
                continue
            for prefix, canonical in _CANONICAL_TOOL_NAMES.items():
                if op_id.startswith(prefix):
                    mcp_names[op_id] = canonical
                    break
    return mcp_names
```

### Server-side: pass `mcp_names` to `FastMCP.from_openapi()`

```python
def create_mcp_server(openapi_spec: dict[str, Any]) -> FastMCP:
    client = httpx.AsyncClient(base_url=DATA_SERVICE_URL, timeout=600.0)
    mcp = FastMCP.from_openapi(
        openapi_spec=openapi_spec,
        client=client,
        route_maps=_build_route_maps(),
        mcp_names=_build_mcp_names(openapi_spec),   # ŌåÉ canonical naming
        name="data-tool-mcp",
    )
    return mcp
```

## Decision Record

- **Decided:** Use `mcp_names` parameter (FastMCP built-in) for server-side canonical naming.
- **Rejected:** `x-displayName` vendor extensions (unnecessary ŌĆö `mcp_names` does it natively).
- **Rejected:** Pattern-based matching in qna-agent (still inferential, can misclassify).
- **Rejected:** Keeping static aliases as primary strategy (highest maintenance and outage risk).
- **Rejected:** Externalizing `_CANONICAL_TOOL_NAMES` to a config file (YAML/JSON). Both sides of the
  mapping are code artifacts ŌĆö keys are Python function names from `@router` decorators, values must
  match system prompts and `_PERSONA_TOOLS` in `registry.py`. Any change to either side already
  requires a code change and redeploy, so a config file would add loading/validation complexity
  without any real operational flexibility. Config files are appropriate for environment-specific
  values (URLs, timeouts, feature flags) ŌĆö not code-to-code contracts.
- **Rejected:** Startup validation of prompt Ōåö `_PERSONA_TOOLS` consistency. Prompts are owned by the DS team and mention tool names in prose ŌĆö parsing them reliably is fragile. `filter_by_persona` is the hard gate; mismatches get caught on first test.
- **Updated (2026-04-02):** `_TOOL_NAME_ALIASES` removed entirely (clean cut). The alias table was
  the root cause of brittleness. If the server-side naming causes issues, the commit can be reverted.

### PR #99 Review Comments (2026-04-06)

**Comment 1 ŌĆö line 39-40 (`_CANONICAL_TOOL_NAMES`):**
> "Do the names from the openapi_spec is same as mentioned here?"

No ŌĆö the keys are stable Python function name **prefixes**, not the full operationIds.
The actual operationIds in the spec are much longer due to route prefix duplication
(e.g. `get_ibat_equipment_endpoint_dataservices_api_v1_ibat_equipment_get`).
`_build_mcp_names()` uses `startswith()` to match them, so the keys here are
deliberately shorter prefixes.

**Comment 2 ŌĆö line 52 (`_build_mcp_names`):**
> "How is the openapi_spec being passed into the method?"

`main.py` lines 81-84:
```python
openapi_spec = temp_app.openapi()
mcp_server = create_mcp_server(openapi_spec)
```
`create_mcp_server()` passes it to both `FastMCP.from_openapi()` and `_build_mcp_names()`.

**Comment 3 ŌĆö line 66-68 (renaming logic location):**
> "Maybe the renaming logic should move there [main.py]. Please do not change tool
> names related to heat_map, ibat data, retriever and er cases."

Two parts:

- **Moving to main.py:** The logic is intentionally in `server.py` ŌĆö it's an MCP server
  concern and `create_mcp_server()` is the only consumer. Keeping all MCP logic
  (route maps, canonical names, server creation) in one module is cleaner.

- **"Do not change tool names":** We are not changing away from the correct names ŌĆö
  we are changing **to** them. The old names were never intentional; they were auto-generated
  artifacts of FastMCP truncation + double prefix. For example, the risk-evaluation-assistant
  was hardcoding `"load_heatmap_dataservices_api_v1_heatmap_load_get"` in
  `risk_assessment_creation.py` ŌĆö PR #107 (by the same reviewer) replaces these with the
  canonical names (`read_risk_matrix`, `query_fsr`, `query_er`). That update is only
  possible because `mcp_names` guarantees those canonical names on the server side.
  The names in `_CANONICAL_TOOL_NAMES` match exactly what system prompts and persona
  allowlists already use.
